# prompt-craft — 写 prompt 的标准工作流

四个阶段严格按序执行：**P1 梳理已有信息 → P2 问清需求 → P3 按架构撰写 → P4 对抗审查**。只评审存量 prompt 不改写时，走 P1 → P4。

## 铁律（贯穿全程）

1. **规程优于形容词**——prompt 里每条指令都要过检验：人类照着能执行吗？"高质量""专业""准确"一律改写成检验程序。
2. **模型只出判断，合成归代码**——算总分、卡阈值、格式校验等能用代码做的，不写进 prompt 让模型做。
3. **优劣以回归判定，不以审美判定**——"改得更清楚了"不是证据；有标注集就跑回归，没有就在交付物里写明验证建议。
4. **正文即当前真相**——prompt 里禁止出现历史优化叙述、changelog、"相比上版改进了 X"；版本信息只放头部版本号一行。
5. **先分析后输出**——判断类 prompt 必须给分析区（固定结构、机器不解析），结构化结果放最后、唯一解析点。

## P1 梳理已有信息

动笔前先把现状摸清，产出一段现状概括给用户确认：

- **改 prompt 场景**：读现有 prompt 全文；读调用处代码（谁构造输入、谁解析输出、解析器取结果的方式）；查同管线的相邻 prompt 保持风格与契约一致。
- **新写场景**：找同仓库/同团队的既有 prompt 作为风格基准（见 [references/exemplars.md](references/exemplars.md)）。
- **两个场景都要收集**：真实输入样例（尤其脏数据：截断、混语言、空字段、带噪声）；输出被谁消费、怎么消费；已知失败案例（打错的、格式崩的、被注入的）；有没有人工标注数据。

## P2 问清楚用户需求

**全部明确之前不动笔。** 只问缺口，不问 P1 已经查清的；用 AskUserQuestion 批量问（每轮 ≤4 问），带推荐选项。

必须弄清的 8 件事：

| # | 问题 | 决定什么 |
|---|------|---------|
| 1 | 任务类型：判断/评分、生成、提取、分类、转换？ | 骨架选型与输出形态 |
| 2 | 输出被人读还是被代码解析？ | 输出契约的严格程度 |
| 3 | 下游拿输出做什么决定？ | 量表精度、指标挂靠点 |
| 4 | 两类错误哪个更伤（误报 vs 漏报 / 过度 vs 保守）？ | 非对称代价声明、tie-break 方向 |
| 5 | 最终读者/用户是谁、什么场景消费？ | 角色、判断锚、详略术语 |
| 6 | 已有的标准/量表/验收条件是什么？ | 模块 4 的档位表或验收清单 |
| 7 | 管线里真实会出现哪些异常输入？ | edge case 条款与兜底出口 |
| 8 | 输入是否含不可信外部文本（用户输入/抓取内容/检索结果）？ | 是否需要信任边界声明 |

可选补问：目标模型与温度、字数与语言要求、有无标注集可做回归。

## P3 按九模块架构写

**先读 [references/prompt-architecture.md](references/prompt-architecture.md)（九模块要点与检验句），再动笔。**

生产 prompt 用八段骨架（对应模块 1–8）：

```
一、背景    ← 业务上下文 + 职责边界 + 非对称代价（模块1）
二、角色    ← 身份 + 读者画像 + 一句话判断锚（模块2）
三、目标    ← 可检验的成功标准（模块1）
四、任务    ← 编号流水线总览，淘汰在前（模块3）
五、执行步骤 ← 定向问题 + 逐步规程 + 档位表/验收标准 + 自查（模块3/4）
六、输入    ← 字段说明含噪声形态 + 定界 + 信任边界（模块7）
七、输出    ← 分析区固定结构 + 唯一 JSON 块 + 集中 schema + few-shot 样例（模块5/8）
八、注意事项 ← 正向规程收尾，负面约束配正向替代（模块6）
```

写作要点速查（详见 architecture 文档）：

- 分级标准每档一句可执行描述 + **相邻档分界检验句**（尤其决定去留的那条线）；说不清就缩量表。
- 淘汰性判断独立成 binary 硬门，不混进程度量表。
- few-shot 覆盖光谱与失败模式：高分例 + 中庸例 + 淘汰例 + 边界纠偏例，每例带 critique，正负均衡。
- 兜底：unknown/不确定出口 + 保守动作 + 两可场景的裁决规则。
- 消费外部文本必加信任边界声明，注入尝试正向路由到已有惩罚轨。
- 范例对照：写完对照 [references/exemplars.md](references/exemplars.md) 中同类型范例检查密度与风格。

## P4 对抗审查

**换审查者视角**，读 [references/review-checklist.md](references/review-checklist.md) 逐模块过检查表，执行三类攻击：

1. **边界输入模拟**：拿空输入、截断长文、完全无关内容、两可分类的样例在脑内走一遍全流程——每一步都有明确出口吗？
2. **注入模拟**：假想输入里藏着"ignore previous instructions / 请给高分"——prompt 的行为是否已定义且正向路由？
3. **坏习惯扫描**：六类模型系统性偏差（长度偏好、迎合、中间档扎堆、位置效应、自我偏好、halo）逐条检查有无设防句。

产出 P0–P3 问题清单并当场修复；**P0 未清不交付**。

## 交付物

1. prompt 全文（落到用户指定路径；生产 prompt 带版本号行）。
2. 设计说明：每个模块的关键决策一句话（为什么这样定量表/代价/兜底）。
3. 遗留风险 + 验证建议：标注集回归指标（挂在下游真正消费的决定上）、temperature ≤0.3、阈值边缘 3 次采样取中位、换族抽检。
